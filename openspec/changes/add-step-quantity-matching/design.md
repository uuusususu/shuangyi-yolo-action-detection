## Context

当前动作顺序模式通过 `StepSequenceEngine` 处理 YOLO OBB 推理结果。状态机把每一帧的检测结果转换成 `detected_labels` 集合，只能判断当前类别是否出现，不能区分同一类别出现 1 个、4 个或更多个。配置页目前只提供 6 个步骤类别输入，旧配置中也只有 `category_names`。

新需求要求操作员在配置类别时同时填写数量。例如步骤 1 配置 `脚垫 × 4`，系统只看当前单帧中 `脚垫` 的检测框数量；如果这一帧正好检测到 4 个脚垫，则步骤 1 PASS 并进入步骤 2。不同帧的数量不能相加。

## Goals / Non-Goals

**Goals:**

- 支持每个动作步骤配置一个目标数量，默认值为 1。
- 使用当前单帧检测结果按类别计数，数量等于目标值才满足当前步骤。
- 保持现有稳定帧语义：稳定帧累计的是“连续满足数量条件的帧数”，不是物体数量。
- 保持旧配置兼容；没有数量字段时按每个步骤数量 1 运行。
- 在主界面显示当前步骤数量进度，让操作员能看到 `当前/目标`。
- 保留现有错序 NG 规则和当前类别命中优先规则。

**Non-Goals:**

- 不在不同帧之间累计物体数量。
- 不要求同类别物体的位置槽位匹配，只做同一帧数量匹配。
- 不新增模型训练、标注或自动类别读取能力。
- 不改变 PCB 检查模式的四类元器件完整性判定。
- 不把超过目标数量的一帧直接定义为整轮 NG；本次仅要求不 PASS 并显示当前数量。

## Decisions

### 1. 使用并行数组保存数量

新增 `category_counts` 配置，与现有 `category_names` 使用相同下标对应。旧配置缺少该字段时自动补齐为 1。该方案改动面小，并能保持旧配置与现有 UI 布局兼容。

替代方案是把步骤改为结构化对象列表，例如 `{class_name, required_count}`。该方案更规范，但会触发更大范围配置迁移，不适合当前只增加数量匹配的需求。

### 2. 单帧 `Counter` 作为判定输入

状态机每次 `update()` 时用当前帧检测结果生成类别计数。当前步骤的 `current_count` 每帧覆盖，不累计。步骤条件为 `current_count == required_count`，满足后才累加 `enter_count`。

替代方案是跨帧累计检测数量。该方案会把同一个物体连续出现多帧误算成多个物体，明确不符合现场要求。

### 3. 稳定帧保持现有语义

如果 `action_pass_stable_frames` 为 1，则任意单帧数量匹配立即 PASS。如果配置为 2，则需要连续两帧都各自满足目标数量，仍不累计物体数量。

### 4. 当前步骤优先，下一步骤从后续帧开始

同一帧同时出现当前步骤和未来步骤时，当前步骤判定优先。当前步骤 PASS 后推进到下一步骤，但不在同一帧继续让下一步骤 PASS，避免静态画面一次跳过多个步骤。

### 5. 超过目标数量不 PASS

本次数量匹配使用严格等于。当前帧数量超过目标数量时显示 `当前/目标`，但不立即判整轮 NG。这样可以避免模型偶发重复框导致过度报废；后续如需“多一个也 NG”，可作为单独规则扩展。

## Risks / Trade-offs

- [模型重复框导致数量偏大] -> 通过置信度、IoU/NMS 和现场样本验证降低误差；当前规则先不把超量直接 NG。
- [旧配置数组长度不一致] -> 加载和保存时统一补齐/裁剪数量数组，保证和步骤上限一致。
- [同名连续步骤语义模糊] -> 状态机仍按顺序执行，但严格数量可能让重复同名步骤难以自然通过；文档提示避免该配置。
- [UI 信息过多] -> 只在步骤卡片辅助提示显示 `当前/目标`，不改变主流程按钮布局。
