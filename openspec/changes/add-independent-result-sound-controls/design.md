## Context

当前 `ConfigManager` 只有 `sound_feedback_enabled`，配置页也只有一个“声音反馈”开关。`MainWindow` 的 PASS 和 FAIL 分支都读取这个字段，因此两类声音无法独立控制。普通顺序模式已经使用 `round_id` 避免同一终局状态重复记数；首类别模式则允许同一父类 ID 在 FAIL 后重新入队，但当前引擎按异常签名抑制整个结果事件，导致同一 ID 的新轮次再次出现相同数量异常时无法产生新的声音。

本变更需要同时覆盖普通顺序模式与首类别多 ID 模式。声音资源和 `SoundFeedback` 播放器已经存在，不需要引入新依赖。现有首类别提案中“相同异常签名不重复播放声音”的描述被本变更取代；异常签名仍可用于统计和证据去重，但声音必须按识别轮次判定。

## Goals / Non-Goals

**Goals:**

- 提供可独立保存和即时应用的 PASS、FAIL 声音开关。
- 默认关闭 PASS 声音、开启 FAIL 声音。
- 兼容只包含旧统一声音字段的配置。
- 普通顺序模式和首类别模式都严格做到一个结果轮次最多播放一次。
- 同一父类 ID 重新入队后的新轮次具有新的声音身份，相同异常也可以再次播放一次。
- 保持声音失败不影响检测主流程。
- 将声音轮次去重与统计、证据、红框的既有语义分离。

**Non-Goals:**

- 不增加音量、音频文件选择、循环播放或播放间隔配置。
- 不改变 PASS/FAIL 稳定帧数、父类 ID 调度、红框持续或补齐后 PASS 规则。
- 不改变异常统计和证据保存的既有去重策略。
- 不删除 PASS/FAIL 声音资源，也不修改迈德相机 SDK、模型或推理线程。

## Decisions

### 1. 使用两个独立布尔配置，不保留 UI 总开关

新增 `pass_sound_enabled: bool = False` 与 `fail_sound_enabled: bool = True`。配置页“操作反馈”区域显示“PASS 提示音”和“FAIL 提示音”两个开关，分别绑定对应字段。

选择两个开关是因为操作员可以直接看到两类结果的真实状态。备选方案“总开关 + 两个子开关”会产生总开关关闭但子开关开启的歧义；单一下拉框虽然能表达四种组合，但扫描和切换成本高于两个二元开关。

### 2. 旧字段只用于迁移，新字段是运行时唯一来源

加载 JSON 时先检查两个新字段是否存在：

- 两个新字段均不存在时，`pass_sound_enabled` 保持 `False`，`fail_sound_enabled` 读取旧 `sound_feedback_enabled`，旧字段缺失时使用 `True`。
- 任一新字段存在时，存在的新字段使用保存值，缺失的新字段使用其安全默认值，不再让旧字段覆盖明确的新配置。
- 保存时写入两个新字段。为兼容可能读取旧配置的旧版程序，可保留旧字段并同步为 `fail_sound_enabled`，但当前运行时不得再用旧字段决定 PASS 或 FAIL 声音。

这样迁移后默认行为就是 PASS 静音、FAIL 按原开关状态报警，避免升级后突然恢复 PASS 声音。

### 3. 播放器保留资源能力，业务分支决定是否调用

`SoundFeedback.play_pass()`、`play_fail()` 和两个声音资源继续保留。播放器的 `enabled` 只作为“至少一种声音开启”的底层能力开关，可设置为 `pass_sound_enabled or fail_sound_enabled`；业务层在调用前必须检查具体结果开关。

不把两个业务开关塞进 `SoundFeedback`，因为播放器只负责安全播放资源，不应知道轮次、父类 ID 或 PASS/FAIL 状态机。

### 4. 使用明确的结果轮次键进行声音去重

普通顺序模式使用 `("action", round_id)` 作为结果轮次键。一个 `round_id` 首次进入 PASS 或 ACTION_NG 时处理一次，后续状态刷新不再播放。

首类别模式不能只使用 `track_id`，因为同一父类 ID 会在 FAIL 后重新入队。引擎应在每次父类被选为新的 `ACTIVE` 识别轮次时分配单调递增的 `attempt_id`，并把它包含在结果事件中。声音轮次键为 `("region", track_id, attempt_id, result)`：

- 同一轮连续达到稳定帧后只产生一个结果身份。
- 重复帧或重复处理同一结果不会再次播放。
- 同一 `track_id` 再次进入 `ACTIVE` 时获得新的 `attempt_id`，新轮次 FAIL 可以再次播放一次。
- PASS 仍是父类终局，只会有一个 PASS 结果轮次。

备选方案使用 `source_frame_id` 去重会把采集帧身份误当成业务轮次，重复回调或回放时不稳定；使用时间防抖无法严格表达“一轮一次”，因此不采用。

### 5. 拆分声音事件与异常签名去重

首类别引擎需要让每个完成的 FAIL 轮次向 UI 暴露轮次结果，即使该异常签名以前出现过。结果中同时携带 `attempt_id` 和“异常签名是否首次上报”的标志或等价信息：

- 声音依据 `(track_id, attempt_id)`，每个 FAIL 轮次一次。
- 统计与证据依据既有异常签名规则，避免本变更无意改变生产数据口径。
- 红框依据父类显示状态和当前可见性，不依赖声音是否播放。

这会取代现有“异常签名直接阻止整个结果进入事件队列”的耦合方式。若实现中已有更小的等价边界，可以不限定字段名称，但必须保留三类反馈的独立语义。

### 6. 保存配置后只更新反馈能力，不重建相机

配置保存后，`MainWindow` 立即读取两个新开关并更新播放器底层 `enabled` 状态。该操作不关闭相机、不重建推理处理器，也不清除父类追踪状态。之后产生的新结果使用新开关；已经播放过的结果不因开关切换而补播。

## Risks / Trade-offs

- [同一父类持续缺件会在每个新轮次再次报警] -> 这是明确的“每轮 FAIL 一次”语义；报警节奏仍由 FAIL 稳定帧数和轮次间隔共同限制。
- [仅按 `track_id` 去重会吞掉重试轮次声音] -> 为每次 ACTIVE 轮次分配 `attempt_id`，声音使用复合轮次键。
- [解除异常签名对结果队列的抑制会重复统计或证据] -> 结果事件携带独立的异常首次上报信息，统计/证据继续使用原规则。
- [旧字段和新字段同时存在产生歧义] -> 新字段优先，旧字段只在两个新字段都缺失时参与迁移。
- [运行中切换开关导致补播旧结果] -> 只对保存后首次产生的新结果应用开关，不扫描或重放历史结果。
- [现有首类别 OpenSpec 与新声音规则冲突] -> 在实现验证和归档前同步修订其声音描述，保留统计/证据的异常签名去重。

## Migration Plan

1. 先增加配置默认值、旧字段迁移和保存/加载测试，固定升级行为。
2. 增加普通顺序和首类别模式的声音 spy 测试，覆盖开关组合、重复回调、同 ID 新轮次及不同 ID。
3. 增加独立配置字段和配置页开关，保存后即时应用。
4. 为首类别结果补充轮次身份，并拆分声音轮次事件与异常签名上报标志。
5. 调整 MainWindow 的 PASS/FAIL 播放分支，确保每个结果轮次最多一次。
6. 更新 README，并同步修订 `add-first-category-region-check` 中与本变更冲突的声音描述。
7. 运行定向测试、完整 pytest、OpenSpec 严格校验；如用户要求交付打包，再执行打包与 packaged smoke。

回滚时可恢复单一 UI 开关和旧运行时分支；两个新字段留在 JSON 中不会被旧版识别，旧 `sound_feedback_enabled` 同步 FAIL 值可保证异常声音偏好尽量保留。
