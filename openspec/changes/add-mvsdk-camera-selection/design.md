## Context

项目内置迈德 SDK Python 封装 `python_demo/mvsdk.py`。`CameraEnumerateDevice(MaxCount=32)` 返回当前在线设备的 `tSdkCameraDevInfo` 列表；设备结构提供 FriendlyName、PortType、唯一序列号 `acSn` 和同型号实例号 `uInstance`。SDK 推荐从枚举结果中选定设备结构后调用 `CameraInit(dev_info, -1, -1)`，关闭时调用 `CameraUnInit()`，应用分配的对齐帧缓冲还必须调用 `CameraAlignFree()`。

当前 `MvSdkCamera.enumerate_devices()` 已把 FriendlyName、PortType 和 SN 转成 `MvSdkDevice`，但 `open()` 只按 FriendlyName 匹配，而且匹配失败会打开第一台。`CameraWorker` 在配置为空时同样选择第一台。配置页没有设备选择控件；关闭相机只释放 SDK 句柄，没有清空工作线程缓存帧和主界面最后一帧、overlay、步骤及结果状态。

本变更跨越 SDK 封装、采集线程、配置持久化和 PySide6 界面。现场操作员需要在多台同型号相机中稳定选择指定设备，并能确认切换后画面和检测状态只属于新相机。

## Goals / Non-Goals

**Goals:**

- 每次程序启动从 SDK 获取当前在线设备，不持久化枚举列表。
- 在配置页“工业相机”分区显示全部在线迈德相机并允许手动刷新。
- 只持久化所选相机 SN，按 SN 从最新枚举结果中精确打开设备。
- 点击“保存”后才应用选择变化；相机正在运行时安全关闭旧设备并预览新设备。
- 相机关闭或切换时清除旧帧、检测框、步骤、父区域跟踪和 PASS/FAIL 结果，但不清零生产统计。
- 对 SDK 不可用、无设备、SN 不在线、设备占用和初始化失败提供明确反馈。
- 保持旧配置可读取，并让能够唯一匹配的旧 FriendlyName 平滑迁移到 SN。

**Non-Goals:**

- 不同时采集或拼接多台相机画面；任一时刻只打开一台相机。
- 不实现相机热插拔自动切换；设备变化由启动枚举或手动刷新发现。
- 不持久化枚举索引、设备列表、SDK 设备结构或相机在线状态。
- 不在本变更中改变曝光、参数组、参数文件和模型推理参数的业务语义。
- 不在切换后自动恢复检测；新相机先进入预览，由操作员重新开始检测。

## Decisions

### 1. SN 是持久身份，枚举索引只用于本次扫描

新增配置字段 `mvsdk_camera_sn`，下拉项显示 `FriendlyName | PortType | SN`，其 item data 保存 SN。`index` 仅用于显示顺序和诊断，不写入 JSON。打开相机时再次调用 `CameraEnumerateDevice()`，比较 `GetSn()`，并将匹配到的原始 `tSdkCameraDevInfo` 交给 `CameraInit()`。

不选择 FriendlyName 作为主键，因为同型号设备可能重名；不选择枚举索引，因为接线顺序、驱动发现顺序和重启都可能改变索引。若 SDK 返回空 SN，该设备仍可显示诊断信息，但不能作为可保存的稳定选择，界面提示设备身份无效。

### 2. 设备目录只驻留内存，并支持启动枚举与手动刷新

`MainWindow` 初始化相机工作对象后执行一次设备枚举，将 `list[MvSdkDevice]` 作为当前会话设备目录保存在内存。首次创建或再次打开配置页时把最新目录传给 `ConfigPage`；“刷新设备”按钮通过信号请求主窗口重新枚举，再调用配置页的列表更新方法。

不把设备列表写入 `config.json`。这样离线设备不会因为历史缓存继续显示为可用。同步枚举保持实现简单；SDK 缺失或枚举异常必须被捕获并显示，不得导致程序启动失败。

### 3. 配置页只负责选择和保存，主窗口负责运行时切换

`ConfigPage` 不直接引用 `CameraWorker` 或 SDK 句柄。配置页负责：渲染设备列表、按已保存 SN 预选、显示离线提示、发出刷新请求，并在现有 `_on_save()` 中把有效选择写入 `mvsdk_camera_sn`。现有 `config_saved` 信号继续作为保存成功后的运行时应用入口。

`MainWindow._on_config_saved()` 比较目标 SN 与 `CameraWorker.active_device.sn`：

- 相机未打开：只更新持久配置，不自动打开设备。
- 相机已打开且 SN 未变化：保持当前采集，不重启相机。
- 相机已打开且 SN 变化：执行完整切换流程。

这种信号边界延续现有“配置页保存、主窗口应用运行时配置”的结构。备选方案是让配置页直接关闭和打开相机，但会把 SDK 生命周期耦合到表单组件，难以处理窗口退出和测试，因此不采用。

### 4. 保存后切换采用“停止检测 -> 停线程 -> 释放 -> 清理 -> 精确打开”

切换顺序固定为：

1. 停止检测、显示定时器和本轮结果定时器，不自动恢复检测。
2. 将 `CameraWorker.running` 置为 false，并等待采集线程退出。采集读帧超时为 200ms，使用有界等待确保线程不再访问 SDK 句柄或帧缓冲。
3. 调用 `CameraUnInit()`，再调用 `CameraAlignFree()`，清除 worker 中的相机引用。
4. 在锁保护下清空 preview、infer、display、overlay、帧 ID 和 pipeline stats 缓存；主窗口清空 `QLabel.pixmap()`、overlay 字典、结果文案、异常提示、步骤状态及父区域跟踪引擎。
5. 使用目标 SN 重新枚举并精确匹配，调用 `CameraInit()`、设置输出格式和参数、`CameraPlay()`、分配新帧缓冲并启动新采集线程。
6. 打开成功后恢复 33ms 显示定时器并进入“预览中”；检测保持关闭。

不采用“先打开新相机再关闭旧相机”，因为相同总线带宽、设备占用和单例 SDK 资源可能导致初始化失败，并且会让两个采集线程短暂并存。

### 5. 切换失败保持关闭，不静默回退

若目标 SN 在最新枚举结果中不存在，或 `CameraInit()` 抛出 `CameraException`，系统保持相机关闭和空白画面，提示 SN 及可用的 SDK 错误码/原因。系统不得回退到第一台、旧相机或 OpenCV `camera_index`。

备选方案是失败后重新打开旧相机，但这会让界面已保存的选择与实际采集设备不一致，现场容易误判，因此不采用。操作员可刷新列表、重新选择并保存，或重新选择旧相机。

### 6. 启动选择和旧配置迁移

启动枚举后：

- `mvsdk_camera_sn` 在线：默认选中对应下拉项。
- `mvsdk_camera_sn` 不在线：不改写配置，显示“已保存相机不在线，请重新选择”。
- 新 SN 字段为空但旧 `mvsdk_friendly_name` 恰好匹配一台在线设备：配置页预选该设备，下一次保存写入其 SN。
- 旧 FriendlyName 匹配零台或多台：不自动选择，要求操作员按 SN 明确选择。
- 两个字段都为空：显示“请选择相机”，不再默认第一台。

为便于旧版本回滚，暂时保留 `mvsdk_friendly_name` 字段；保存新选择时可同步写入所选设备的 FriendlyName，但运行时选择权以 SN 为准。

### 7. 关闭与切换复用统一的视觉会话清理

主界面“关闭相机”、保存后切换、窗口退出前的运行态清理复用同一条相机会话关闭路径。普通关闭和切换清除检测会话状态但不影响总数、OK、NG、良率和批次记录。窗口退出无需更新可见控件，但必须使用相同的线程停止与 SDK 释放顺序。

## Risks / Trade-offs

- [SDK 枚举在 UI 线程耗时] -> 启动和手动刷新各执行一次并捕获异常；若现场实测明显阻塞，再独立引入异步枚举，不在本次预先增加线程复杂度。
- [采集线程仍在 `CameraGetImageBuffer()` 时释放句柄] -> 先停止并有界等待采集线程退出，再执行 `CameraUnInit()` 和 `CameraAlignFree()`；针对调用顺序增加单元测试。
- [保存配置成功但新相机初始化失败] -> 保留已保存 SN，运行时保持关闭和清屏，显示具体失败原因；不让实际相机与配置身份不一致。
- [同名相机导致旧配置迁移歧义] -> 只有 FriendlyName 唯一匹配时才预选，否则必须人工按 SN 选择。
- [设备在枚举后、保存前被拔出] -> 打开时再次枚举并按 SN 匹配；失败路径保持关闭，不使用过期下拉项对应的索引。
- [切换后旧父区域 ID 或步骤结果继续存在] -> 重建或显式重置步骤/区域检查引擎，并清空全部 overlay、结果和帧缓存。

## Migration Plan

1. 新增默认空值 `mvsdk_camera_sn`，加载旧 JSON 时继续接受 `mvsdk_friendly_name`。
2. 发布后首次启动重新枚举设备；若旧名称唯一匹配则在配置页预选，但只有操作员保存后才持久化 SN。
3. 对已有空配置不再自动选择第一台，操作员需要在“工业相机”分区完成一次明确选择与保存。
4. 回滚旧版本时，旧程序会忽略未知的 `mvsdk_camera_sn`，仍可读取同步保留的 `mvsdk_friendly_name`。

## Open Questions

无。设备列表不落盘、SN 持久化、保存后切换、切换失败保持关闭以及切换后不自动恢复检测均已在本提案中确定。
