## Context

当前程序启动时会根据配置创建一次 YOLO/ONNX 推理处理器，并把它设置给相机推理线程。配置页保存后会修改配置对象，但运行中的处理器不会重新加载；配置文件也主要依赖程序退出时保存。现场选择新模型后，界面容易出现“配置看起来已保存，但推理仍使用旧模型或没有可用模型”的问题。

业务步骤类别由操作员在配置页手动填写。模型类别列表只能作为推理输出来源，不能自动覆盖业务配置。状态机仍然按模型返回的 `detection.label` 与手动配置的 `category_names` 做业务判断。

## Goals / Non-Goals

**Goals:**
- 保存配置后立即应用运行时配置，不需要重启程序才能使用新模型。
- 模型路径或推理参数变化时，重新加载推理处理器并替换相机推理线程使用的处理器。
- 保持业务逻辑不变：模型返回类别名称与手动配置类别名称匹配后才推进步骤。
- 配置类别始终由操作员手动维护，不从模型类别自动填充或覆盖。
- 新模型加载失败时保留旧处理器，并给出明确错误提示。

**Non-Goals:**
- 不改变 PASS/NG 顺序状态机规则。
- 不新增从模型类别自动生成步骤配置的能力。
- 不改变训练、导出或模型格式要求。
- 不重新设计配置页整体 UI。

## Decisions

### 1. 配置页只发出保存事件，主窗口负责应用运行时配置

配置页继续负责收集并校验表单值。保存成功后发出 `config_saved` 信号，由主窗口统一处理模型重载、步骤引擎重建、worker 替换和状态提示。这样可以避免配置页直接依赖推理器、相机线程和主窗口内部状态。

替代方案是在配置页里直接重载模型。该方案会让配置页承担过多运行时职责，后续维护风险更高，因此不采用。

### 2. 模型处理器使用“先加载成功，再原子替换”策略

当模型路径或推理参数变化时，主窗口先暂停推理，使用当前配置创建新处理器并调用加载。只有加载成功后，才替换 `self.processor` 和 `worker.frame_processor`。如果加载失败，保留旧处理器，并提示错误。

替代方案是先清空旧处理器再加载新模型。该方案失败时会让现场进入无模型状态，因此不采用。

### 3. 手动配置类别是业务判断唯一来源

配置页保存时不会读取模型类别并写入 `category_names`。状态机仍以手动配置的步骤类别为准。模型推理返回的类别只作为检测结果输入，业务判断使用 `detection.label == 当前步骤类别名`。

可以显示模型类别作为只读诊断信息，但本次变更不要求增加自动同步或自动填充按钮。

### 4. 复用现有模型创建入口

主窗口重载模型时复用 `main.create_yolo_processor(config, app_base_dir)`，避免在 UI 层重复维护 `.onnx` 和 `.pt` 两套加载逻辑。

## Risks / Trade-offs

- [模型加载耗时导致短暂停顿] -> 加载期间停止推理并显示状态，避免推理线程并发访问旧/新处理器。
- [新模型类别与手动配置不一致] -> 业务逻辑保持不变，不自动修正；后续可用只读提示帮助操作员排查。
- [保存失败导致配置和运行态不一致] -> 保存配置文件失败时提示错误，不报告配置保存成功。
- [运行中换模型时相机仍打开] -> 相机预览可以保持，推理线程暂停后再恢复，减少现场操作成本。
