## Context

当前相机线程采用“追最新帧、丢旧帧”的方式，把每个待推理画面交给同一个 YOLO OBB 处理器。单次结果已经包含类别、置信度、OBB 多边形、中心点和可选跟踪 ID，但现有业务状态机把检测类别解释为作业步骤，尚未建立“PCB 产品—板内元器件”的父子关系。

新场景中，多块 PCB 沿同一流水线运动并保持间隔。模型在完整画面中同时输出 `pcb` 和四种位置固定、类别名称互不相同的元器件。现场要求每个画面最多执行一次模型推理，仅持续跟踪 PCB；元器件不依赖持续 ID，而是在每次检测结果中重新归属到 PCB。缺件判断必须容忍短暂漏检，已完成产品不得重复计数。

## Goals / Non-Goals

**Goals:**

- 复用一个模型和一次完整画面推理，同时更新画面中所有 PCB 的检查状态。
- 以 PCB 跟踪 ID 作为产品身份，把四种元器件稳定归属到对应 PCB。
- 在任意有效帧完整时尽快 PASS，在连续三个有效帧不完整时 FAIL。
- 将完成判定与结果输出解耦，使轮次间隔不阻塞采集、推理和其他 PCB 的观测。
- 复用现有 UI、证据、声音和生产统计能力，并保留原作业顺序模式。

**Non-Goals:**

- 不增加第二个模型，不对每块 PCB 再执行裁剪推理。
- 不持续跟踪元器件，也不把元器件 tracker ID 作为业务身份。
- 不建设模型训练、数据标注或自动槽位标定工具。
- 不处理 PCB 相互接触、重叠或严重遮挡的场景。
- 本次仅判断四个配置类别是否齐全，不判断焊接质量、轻微姿态偏差或未配置的额外元器件。

## Decisions

### 1. PCB 检查作为独立可选业务模式

新增 `pcb_inspection_enabled` 配置。启用时，推理结果进入独立的 `MultiPcbInspectionEngine`；未启用时，现有作业顺序状态机行为保持不变。这样可以复用相机和模型运行时，同时避免对现有交付行为造成隐式破坏。

备选方案是直接替换现有步骤状态机，但这会让原有动作顺序场景失效，因此不采用。

### 2. 一个推理结果同时服务所有 PCB

每个待推理完整帧只调用一次现有处理器。引擎从同一个 `DetectionOverlayState` 中筛出 PCB 和四种配置元器件，并并行更新所有 PCB 状态，不为单块 PCB 排队调用模型。

备选方案是先检测 PCB，再把每块 PCB 裁剪后送入同一模型。该方案在同屏 PCB 数量增加时会产生 `1 + N` 次推理，不符合当前吞吐目标，因此作为未来精度不足时的退路，不纳入本次实现。

### 3. 只跟踪 PCB，元器件使用临时归属

只有带有效 `track_id` 的 PCB 才建立产品状态。元器件检测携带的 tracker ID 即使存在也被忽略，其业务身份定义为 `(pcb_track_id, component_class)`。四个配置类别分别代表四个逻辑槽位；同一类别出现多个检测时只取归属得分最高、置信度最高的一个作为该槽位的本帧观测。

元器件归属由独立的 `ComponentOwnershipResolver` 完成：

1. 优先判断元器件中心点是否落在带少量容差扩张的 PCB OBB 多边形内。
2. 使用“元器件多边形与 PCB 归属区域的交叠面积 / 元器件面积”作为归属得分。
3. 仅有一个合格 PCB 时直接归属；存在多个候选时选择得分最高且与第二名具有安全差值的 PCB，否则将该检测视为未归属。
4. 未归属检测不能给任何 PCB 槽位提供“存在”证据；若归属歧义的候选包含某块 PCB，则该 PCB 的本帧观测标记为不可靠，不累计 FAIL。

相比同时跟踪 PCB 和元器件，该方案不会因为小目标漏检、ID 重建或相邻板上同类目标造成业务串 ID。

### 4. 仅在 PCB 完整可检查时累计 FAIL

PCB 必须具有有效跟踪 ID，且 OBB 多边形完整位于画面安全边距内，才视为一个有效检查帧。刚进入或即将离开画面的不完整 PCB 保持等待状态，不累计缺件帧，避免板外区域尚未进入画面就被判 FAIL。

每个 PCB 保存独立状态：`OBSERVING -> DECIDED -> EMITTED -> RETIRED`。在有效检查帧中：

- 四个逻辑槽位全部存在时立即生成 PASS 决策。
- 任意槽位缺失时，连续 FAIL 计数加一。
- 连续 FAIL 计数达到配置值时生成 FAIL 决策；默认值为 3。
- PCB 不完整，或存在与该 PCB 相关的归属歧义时，本帧不增加计数；没有对应类别检测且不存在歧义时仍属于有效缺件证据。
- 一旦生成最终决策，该 PCB 立即停止继续判断，后续同 ID 检测只用于显示和离场管理。

### 5. 并行判定与串行结果间隔解耦

所有 PCB 状态在每次推理后并行更新。最终决策进入按 PCB 首次出现顺序排列的结果队列，结果事件之间使用 `pcb_round_interval_seconds` 控制最小间隔。间隔期间相机采集、单模型推理、PCB 跟踪和未完成 PCB 的观测继续运行；已经决策但尚未输出的 PCB 也不会再次判断。

该设计比“等待间隔时停止推理”更适合连续流水线，也比“每次只观察一个 PCB”更少浪费有效帧。间隔配置为 `0` 时，已就绪结果可以连续输出。

### 6. 配置模型和校验

新增以下核心配置：

- `pcb_inspection_enabled`：是否启用 PCB 检查模式。
- `pcb_class_name`：PCB 模型类别名称。
- `pcb_component_class_names`：恰好四个非空、互不重复的元器件类别名称，顺序同时作为逻辑槽位顺序。
- `pcb_fail_stable_frames`：连续不完整帧阈值，默认 `3`，最小值 `1`。
- `pcb_round_interval_seconds`：结果轮次最小间隔，默认 `0.0`，不得小于 `0`。
- `pcb_assignment_margin_ratio`：PCB 归属区域扩张比例，用于容纳位于板边的插接件，提供保守默认值并做范围校验。

操作员在现有配置页填写类别、FAIL 帧数和轮次间隔。类别不完整、重复或与 PCB 类别冲突时，系统阻止启动 PCB 检查并显示明确错误，不能静默退化为错误判定。

### 7. 模块边界和数据流

新增 `src/pcb_inspection/`，保持纯业务逻辑与 UI、相机 SDK 解耦：

- `models.py`：配置快照、PCB 状态、槽位观测和最终结果数据结构。
- `geometry.py`：OBB 扩张、点包含、交叠得分和唯一归属计算。
- `engine.py`：多 PCB 生命周期、连续帧判定、已处理去重和结果队列。

`CameraWorker` 和 YOLO 处理器无需改变推理次数。`MainWindow` 在 PCB 模式下把每次 overlay 交给新引擎，并把引擎结果映射到现有声音、证据和生产统计接口。UI overlay 显示 PCB ID、当前已识别槽位、FAIL 连续帧和最终 PASS/FAIL。

### 8. 跟踪丢失和状态释放

短暂漏检时状态保留一个受控宽限期；同一 ID 恢复后继续累计。若 tracker 在相邻位置生成新 ID，引擎只在运动方向、位置距离和时间窗口均满足且候选唯一时迁移未完成状态。已经完成的 PCB 在离场宽限期内保留去重记录，降低短暂 ID 切换导致重复结果的概率。

PCB 在获得足够有效帧前离开画面时，状态以“未完成”释放，不写入 PASS/FAIL 生产统计；诊断日志记录原因，便于发现视野或速度配置问题。

## Risks / Trade-offs

- [完整画面缩放后元器件像素不足] → 使用真实流水线全图验证模型与 `imgsz`；若仍不足，再单独提出二次裁剪方案。
- [单帧误检导致 PASS] → 四种类别必须同时归属同一 PCB；保留类别置信度阈值，并通过现场缺件测试验证。当前按用户吞吐要求保持 PASS 单帧完成。
- [板边元器件中心在 PCB 外] → 使用有限扩张区域和交叠得分，扩张比例必须受范围约束，避免吸附到相邻 PCB。
- [PCB tracker ID 切换导致重复计数] → 使用短期状态迁移和已完成离场宽限；诊断日志记录 ID 迁移。
- [结果间隔导致后续 PCB 丢失] → 间隔仅限制结果事件，不暂停推理和状态更新；就绪结果进入有界队列。
- [配置类别与模型不一致] → 启动前校验模型类别并给出错误，不开始生产判断。
- [PCB 接触或重叠时空间归属不唯一] → 本次明确要求 PCB 保持间隔；歧义检测不提供存在证据，并输出诊断信息。

## Migration Plan

1. 以默认关闭的方式加入 PCB 检查配置，现有配置文件缺少新字段时继续运行原作业顺序模式。
2. 在测试环境填写 PCB 与四个元器件类别，使用正常板、逐项缺件板和多板视频验证。
3. 验证一帧一次推理、结果去重、生产统计和证据保存后，再在现场启用 PCB 检查模式。
4. 如需回退，关闭 `pcb_inspection_enabled` 或切回原分支；本变更不修改模型文件和历史生产记录格式。

## Open Questions

无阻塞设计问题。具体 PCB/元器件类别名称、归属容差和轮次间隔由操作员在部署时按实际模型与线速填写，并通过验收数据确认。
