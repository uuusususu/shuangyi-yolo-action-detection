## Context

当前桌面端运行在 Python 3.11、PySide6 6.10.1 和 Windows 环境，并通过 PyInstaller 生成便携版目录。主检测页和配置页已经形成稳定的深蓝科技风布局，但交互控件样式由 `runtime_ui_tokens.py`、`config_page.py`、`main_window.py` 和 `native_panels.py` 中多处分散 QSS 共同决定。

现有通用输入样式使用接近背景色的单像素描边；配置页“返回检测”明确取消边框；下拉框和数字框没有完整定义箭头资源；当前开关通过 `QCheckBox::indicator` 绘制胶囊底色但没有标准滑块反馈。这些规则覆盖 Qt 平台样式后，使不同控件的默认、悬停、按下、焦点和禁用状态不一致。

本变更引入外部主题依赖，因此必须同时处理许可、应用级主题初始化、现有 QSS 冲突、无障碍焦点、PyInstaller 资源收集和便携版验收。现场操作员是主要使用者，维护人员需要继续使用现有配置、相机和检测流程。

## Goals / Non-Goals

**Goals:**

- 使用成熟的开源 Qt 主题统一标准控件，不自行绘制按钮、输入框、下拉框、数字框或开关。
- 让所有可执行操作在默认静止状态下即可辨认，并完整覆盖悬停、按下、焦点、选中和禁用状态。
- 保留当前布局、深蓝背景、青色强调色、卡片层级以及 PASS/FAIL/等待/锁定状态语义。
- 保持配置保存、相机 SN 选择、运行时状态、统计、声音和证据行为不变。
- 让源码运行和 PyInstaller 便携版使用同一主题结果，并通过截图与运行验证。

**Non-Goals:**

- 不重新设计主界面或配置页的信息架构、尺寸比例和字段分组。
- 不引入 WebView、QML 或前端浏览器运行时。
- 不修改迈德 SDK 包、相机枚举实现、模型推理或检测状态机。
- 不在本变更中增加亮色主题、主题切换器或用户自定义配色。
- 不把所有状态卡片和检测框改成第三方控件；这些领域状态继续由项目 token 表达。

## Decisions

### 1. 采用 PyQtDarkTheme，而不是 QDarkStyle、qt-material 或 QFluentWidgets

使用 `pyqtdarktheme==2.1.0`（导入名 `qdarktheme`）作为首个经过验证的版本，并保留其 MIT 许可证信息。该版本支持 PySide6，标准 `QPushButton` 默认包含可见描边，输入框、下拉箭头、数字框步进按钮、复选框和键盘焦点状态均由库提供。

不采用 QDarkStyle，因为其标准按钮规则仍使用 `border: none`，不能直接解决本次问题。不采用 qt-material，因为 Material 交互习惯偏向无边框或文本按钮。不默认采用 PySide6-Fluent-Widgets，虽然它提供完整控件类，但 GPLv3 会给闭源外部交付带来授权义务，并额外引入 frameless-window、darkdetect 和 Windows pywin32 打包依赖。

### 2. 主题库负责标准控件，项目 token 负责容器和业务状态

在 `QApplication` 创建后、窗口创建前集中初始化深色主题，使用 `#35D7FF` 作为 primary color。主题库负责 `QPushButton`、`QToolButton`、`QLineEdit`、`QComboBox`、`QSpinBox`、`QDoubleSpinBox` 和 `QCheckBox` 的形状、边界、箭头及交互状态。

现有 `PAGE_BG`、`PANEL_BG`、`STROKE_MAIN`、文字色和步骤状态色继续负责窗口、卡片、相机视窗、识别列表、PASS、FAIL、等待和锁定状态。项目可以保留少量主操作或危险操作的语义属性，但不得重新实现标准控件的边框、箭头或选中图形。

### 3. 删除冲突的通用控件 QSS，不做机械类替换

配置页继续使用 PySide6 原生控件和现有信号/API，因此不会引入 QFluentWidgets `ComboBox` 与 `SwitchButton` 的接口适配风险。删除或停止应用 `input_style()`、`config_button_style()`、`switch_checkbox_style()` 等会覆盖主题库的标准控件绘制规则；保留 `frame_style()`、`text_style()`、步骤状态样式和布局常量。

配置开关使用主题库绘制的标准 `QCheckBox` 指示器。这样 `stateChanged`、`isChecked()`、`setChecked()` 和既有测试保持兼容，也不需要自定义 `paintEvent`。

### 4. 使用语义角色区分操作层级

- 主操作：配置页“验证并保存”和主界面当前关键动作使用主题的 primary/default 表达。
- 次级操作：“浏览”“刷新设备”“返回检测”“配置”等使用清晰描边的普通按钮。
- 危险操作：“关闭”“归零并归档”保留红色危险语义和确认流程。
- 导航操作：配置页六个分区仍使用可选中按钮，未选中时保持可见边界，选中时使用强调背景和焦点状态。

所有文字按钮保持不低于现有 40px 操作高度，主操作保持不低于 52px；不会通过缩小控件来换取边界清晰度。

### 5. 视觉验收使用真实渲染，而不是只检查源码字符串

自动测试负责验证主题已初始化、控件未被局部 `border: none` 覆盖、控件类型与信号行为保持兼容、配置导航与保存正常。视觉验收必须实例化真实 `QApplication` 并渲染主界面和配置页六个分区，在 1366×920、1920×1080 以及 Windows 100%/125% 缩放下检查默认、焦点、选中和禁用状态。

截图验收需确认：不悬停也能识别所有操作控件；下拉箭头和数字步进箭头存在；复选框开关状态明确；文字不裁切；控件不重叠；深蓝/青色科技风和红绿业务状态保持。

### 6. 便携版是最终交付边界

依赖安装后先运行定向 UI 测试和全量测试，再运行源码 `--portable-smoke-test`。随后使用现有 `packaging/build_yolo.bat` 重新打包，验证打包 EXE 返回码为 0、主题资源没有缺失、配置页可打开、控件实际渲染，并记录最终产物路径和验证结果。

## Risks / Trade-offs

- [应用级主题影响非配置页控件] -> 保留项目容器和状态 token，并为主界面、配置页及对话框分别截图验收，发现全局漂移时收窄主题覆盖或提高容器选择器精度。
- [局部 QSS 优先级继续覆盖主题] -> 清点所有 `setStyleSheet()` 和通用控件选择器，只保留容器、文字和业务状态规则；增加禁止标准控件无边框覆盖的回归检查。
- [主题库升级改变视觉] -> 首次实现固定 `pyqtdarktheme==2.1.0`，升级必须单独验证后再修改版本。
- [PyInstaller 漏收集主题模块或资源] -> 在 spec 中显式验证打包 EXE，并在需要时通过 spec 的 hidden imports/data collection 补齐，不以源码成功代替打包成功。
- [标准复选框与现有胶囊开关外形不同] -> 保持行布局、标题、帮助文字和右侧位置不变，以清晰的勾选/未勾选状态优先于装饰性胶囊外形。
- [默认按钮语义被误用] -> 每个页面只设置一个主要默认操作，危险操作永不设置为默认按钮。

## Migration Plan

1. 增加并固定主题依赖，加入集中主题初始化入口。
2. 先以测试覆盖主题初始化和配置页控件行为，再移除冲突的控件 QSS。
3. 依次迁移配置页导航、输入控件、开关和底部操作栏，再统一主界面操作按钮。
4. 运行配置、相机、UI、声音和检测状态相关回归测试及全量测试。
5. 生成多尺寸截图，完成源码 smoke、便携版构建和打包 EXE 验收。
6. 若主题导致无法接受的全局视觉回归，回滚本变更新增依赖和主题初始化，并恢复本变更删除的控件样式；不影响配置文件、模型或业务数据。

## Open Questions

无。默认按 MIT 许可的 PyQtDarkTheme 路线实施；除非后续明确接受 GPLv3 或商业授权，否则不切换到 QFluentWidgets。
