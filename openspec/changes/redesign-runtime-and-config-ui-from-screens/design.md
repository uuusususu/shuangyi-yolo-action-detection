## Context

项目中已有两个 HTML 原型：

- `yolo_action_detection/main-screen.html`：定义实时检测主界面，包含顶部栏、相机视窗、右侧 KPI、识别列表、异常提示和主操作按钮。
- `yolo_action_detection/config-screen.html`：定义系统配置界面，包含左侧分类导航、右侧配置表单和底部固定保存栏。

当前 PySide6 界面已经使用深色科技风 token，但结构仍偏传统：配置页为单页滚动表单，主界面步骤区使用大尺寸 `StepCard`。新设计要求信息层级更接近现场操作：主界面关注实时识别进度和产能，配置页关注分类编辑和明确保存。

## Goals / Non-Goals

**Goals:**

- 将两个 HTML 原型转译为 PySide6 原生控件布局，不引入 WebView。
- 主界面保留现有相机预览、检测按钮、配置入口、KPI 和结果反馈逻辑。
- 主界面右侧改为识别列表，按配置顺序显示已识别类别和数量进度。
- 已配置类别在启动后立即显示，状态变化只更新样式与数量，不改变配置项是否可见。
- 配置页改为左侧导航 + 右侧分区表单，减少滚动和配置混乱。
- 所有表单项保留可见 label、帮助说明和明确保存反馈。
- 保持现有配置字段和业务状态机，不在本次改动中重写检测逻辑。

**Non-Goals:**

- 不引入 HTML 渲染或浏览器 WebView。
- 不改变模型推理、相机采集、统计记录、FAIL 证据保存逻辑。
- 不重新设计训练、标注或数据集管理。
- 不把区域检查的完成间隔重复配置到多个位置。

## Decisions

### 1. 使用 PySide6 原生组件复刻 HTML 原型

HTML 文件作为视觉和信息架构参考，不作为运行时 UI。这样可以保持当前桌面程序部署方式、测试方式和相机渲染链路稳定。

### 2. 主界面改为“实时画面 + 识别进度列表”

保留左侧大相机视窗，右侧从大卡片状态区改为更紧凑的识别列表。识别列表应能显示 `序号 / 类别名 / 当前数量进度或状态`，并在完成一轮间隔期间保持已完成绿色状态。

### 3. 配置页使用导航分区，不再把所有配置堆在同一屏

使用左侧导航按钮控制右侧 `QStackedWidget`。底部固定保存栏提供“放弃更改/验证并保存”，保存成功显示短暂状态提示。现场常用配置优先放在前两个分区。

### 4. 完成一轮间隔只属于动作判定

配置页不在区域检查分区重复提供轮次间隔。普通顺序模式和首类别区域检查模式都使用同一个 `round_cooldown_seconds`。

### 5. UI/UX 约束

- 控件高度不低于 40px，主操作按钮不低于 52px。
- 所有输入必须有可见 label，复杂项必须有 helper text。
- 成功、失败、禁用、悬停/焦点状态要清晰，不只依靠颜色表达。
- 深色背景下文字对比度必须足够，弱信息使用 muted 文本但不低到不可读。
- 每屏只突出一个主要操作，危险操作使用确认对话框。

### 6. 数量配置在两种模式中保持一致

普通顺序模式在首次启动和保存配置后都向步骤引擎传入 `category_counts`。首类别区域检查从同一组类别与数量配置派生父区域和子控件要求；每个子控件按当前帧、当前父区域内的精确数量判定，不跨帧累计。

## Risks / Trade-offs

- [重构配置页可能影响保存字段] -> 保持原 `ConfigPage._on_save()` 语义，重构时补保存/加载回归测试。
- [主界面识别列表与状态机映射复杂] -> 先复用现有 `StepState` 和数量字段，不重写状态机。
- [视觉改动影响现场熟悉度] -> 保留按钮文案和核心位置：配置入口仍在顶部，启动相机/开始检测仍在右下。
- [一次性改动过大] -> 分阶段实现：先主界面结构，再配置页导航，最后视觉 polish 和截图验收。
